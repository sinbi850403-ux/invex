# /tdd — INVEX TDD 워크플로우

**대상**: $ARGUMENTS

테스트를 **먼저** 작성한다. 구현보다 테스트가 항상 먼저다.

---

## RED → GREEN → REFACTOR

### 🔴 RED: 실패하는 테스트 먼저

```javascript
// 예: src/__tests__/inventory.test.js
import { describe, it, expect } from 'vitest';

describe('재고 삭제', () => {
  it('존재하는 품목을 삭제하면 재고 목록에서 제거된다', () => {
    // ...
  });

  it('존재하지 않는 품목 삭제 시 에러를 반환한다', () => {
    // ...
  });

  it('마이너스 수량 입력 시 0으로 보정된다', () => {
    // ...
  });
});
```

테스트 실행 → **반드시 실패해야 함**:
```bash
npm test
```

### 🟢 GREEN: 최소한의 코드로 통과

테스트를 통과시키는 **가장 작은** 코드만 작성.
완벽한 코드가 아니어도 됨 — 일단 테스트만 통과시킬 것.

```bash
npm test  # 모두 PASS 확인
```

### 🔵 REFACTOR: 코드 정리

테스트가 통과한 상태에서 코드 개선:
- 중복 제거
- 변수명/함수명 명확하게
- 상수 추출
- 주석 한글로 정리

```bash
npm test  # 리팩토링 후에도 PASS 확인
```

## INVEX 테스트 커버리지 기준

| 코드 유형 | 목표 커버리지 |
|----------|------------|
| 일반 유틸리티 | 80%+ |
| 재고 계산 로직 | 100% |
| 금액/원가 계산 (`price-utils.js`) | 100% |
| 인증 로직 (`auth.js`) | 100% |
| Supabase DB 함수 (`db.js`) | 80%+ |

## 테스트해야 할 케이스 체크리스트

**Happy Path (정상 동작):**
- [ ] 기본 시나리오 작동

**Edge Cases (경계값):**
- [ ] 빈 값 (빈 문자열, null, undefined)
- [ ] 0, 음수 (수량, 금액)
- [ ] 매우 큰 값 (재고 9999999개)
- [ ] 특수문자 포함 이름

**Error Cases (오류 상황):**
- [ ] 네트워크 오류 (Supabase 연결 실패)
- [ ] 권한 없는 접근 (다른 user_id)
- [ ] 중복 데이터

## 규칙

- ❌ 테스트 없이 코드 작성 절대 금지
- ❌ RED 단계(실패 확인) 건너뛰기 금지
- ✅ 한 번에 하나의 테스트만 작성
- ✅ 테스트가 GREEN이 된 후에만 리팩토링
